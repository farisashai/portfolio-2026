## Why We Left

We migrated our enterprise control plane off Next.js onto a Vite-built static SPA earlier this year. The decision was a long time coming. Next.js had been quietly upselling features that weren't needed for years, and the steady cadence of security advisories and middleware-related incidents in that part of the stack made the long-term posture trade-off harder to justify each quarter. For an authenticated console, almost nothing Next.js was charging in complexity, attack surface, or runtime cost was paying back.

The migration was about that posture first. The performance wins were a known consequence and came along for the ride.

## What an Authenticated Console Actually Needs

An enterprise console is not a marketing site. The premises that justify a Next.js server runtime mostly do not apply:

- No SEO. Crawlers stop at the login page.
- No anonymous traffic worth caching. Every byte is user-specific.
- No first-paint advantage from server rendering, because the user is staring at a shell with skeletons until their authenticated data arrives anyway.
- No content that benefits from ISR or edge caching, because the data is dynamic and per-tenant.

So we asked the inverse question: what *is* the server runtime doing? Auth glue. A middleware layer that re-checks sessions. Image optimization for assets that are mostly icons and logos. A second deployment target for code that exists only to forward requests to the real backend.

The honest answer: it was a backend-for-frontend, and we already had a real backend.

## The Class of Problem We Wanted to Avoid

The cleanest framing is *avoiding classes of problems*, not optimizing marginal gains.

Server-rendering an authenticated route saves a render frame at the cost of an entire category: middleware CVEs, edge runtime quirks, cold starts, region-hop latency, framework-specific auth patterns, and a supply chain that includes the framework's server. Each instance can be patched, or the category can be removed entirely.

Loading skeletons and route-based code splitting solve the user-facing concern without any of those costs. Modern bundlers tree-shake aggressively, and on any modern browser the JS budget for a route-split SPA is not the bottleneck. The API is. Once that was accepted, the server runtime had nothing left to justify.

## Auth, Moved to the Real Backend

Auth used to live in Next.js: middleware verifying cookies, server routes for login/callback/logout, the WorkOS server SDK linked into our frontend deployment.

It now lives entirely in the Go API:

- Login, callback, and logout are HTTP endpoints on the API.
- The session cookie is `HttpOnly`, `Secure`, `SameSite=Strict`, scoped to the registrable domain so the browser still sends it with every request between `app.<domain>` and `api.<domain>` while refusing to attach it to any cross-site context.
- The frontend never reads the session cookie. It calls the API with `credentials: "include"` and lets the browser forward cookies transparently.
- The frontend only knows whether the session is authenticated by what the API returns. There is no JWT in the bundle, no token in localStorage, and no refresh logic running in JavaScript that knows anything secret.

The frontend went from "an auth participant" to "an unauthenticated client that asks the API who the user is." That is the entire model.

## What "Hand Someone the Repo" Means

The cleanest test for this architecture: handing the entire frontend repo to a stranger, including the deployment environment and every build-time variable, would only enable them to rebuild the product. They could not impersonate users, exfiltrate sessions, or pivot into the infrastructure, because the frontend has no credentials to leak. Build-time config is an API origin and a public WorkOS client identifier. Nothing else.

This is a structural property. If a credential is required to operate the frontend in production, the frontend is a privileged participant in the security model. We removed every such credential.

## What We Got Back

### A 20 to 60 percent drop in API response time

This one is measurable and stable across our workloads. Every request from the SPA goes directly from the browser to the API over a single multiplexed HTTP/2 connection. There is no Next.js server in the path. No serialization through an edge runtime, no second TLS handshake, no detour through an external region.

The win compounds: HTTP/2 multiplexing means the second, third, and fourth concurrent request don't pay connection setup, and there is no ephemeral edge function spinning up to validate them. For workloads that fan out a few queries on a page, the practical end-to-end response time drops 20 to 60 percent.

### A static artifact a CDN can serve forever

The build output is HTML, CSS, JS, and assets. It is immutable per release. We host it on a CDN, set aggressive cache headers on hashed assets, and the per-user serving cost is essentially zero. Uptime is the CDN's uptime. There is no frontend service to page on.

### The ability to let customers self-host the control plane

This is the unlock that mattered most for our enterprise roadmap. Because the frontend is a static bundle that talks to an API origin we don't need to control, customers who require a self-hosted control plane can deploy the same artifact behind their VPN, inside their VPC, pointed at their own API. We don't see their traffic. They can audit the entire frontend source against the build output and confirm nothing phones home, because nothing in the bundle has anywhere to phone.

A Next.js application with server runtime, edge middleware, and framework-managed auth could not offer this without a substantial rebuild. A static SPA offers it on day one.

### A smaller, more legible supply chain

We removed Next.js, the WorkOS server SDK, the middleware layer, image optimization, and every transitive dependency they pulled in. What remains is React, a router, a query client, and a UI layer. Every dependency on the runtime path is one a single engineer can read in an afternoon. The server-side dependency tree is now zero.

The CSP, CSRF, and XSS surface area is mitigated through strict `default-src 'self'`, no `dangerouslyInnerHTML` outside a vetted markdown path, and a linter that fails the build if anyone reaches for it. React's default escaping handles the rest. None of this is novel. It just becomes tractable when the surface is small.

## The Migration

The migration was gradual. We removed surfaces in order:

1. Server routes deleted. Auth endpoints moved to the API; the frontend learned to call them like any other endpoint.
2. `next/router` and `nuqs` replaced with TanStack Router. URL state became route-local typed search params validated by Zod schemas; navigation became a typed router instead of a string-based one.
3. `next/link` replaced with TanStack Router's `Link`. Same prefetch ergonomics, no framework-internal route manifest.
4. `next/image` removed. Static assets are pre-optimized at build time or served directly. The console doesn't have hero images that justify a runtime optimizer.
5. CORS configured on the API for the single trusted subdomain that hosts the SPA. SameSite cookies handle the rest.

Each step was independently shippable. There was never a "big bang" cutover.

## What Next.js Is Still Good At

It's worth being precise about where Next.js still earns its keep, because most of the criticism above is downstream of one specific shape of application. The framework hasn't stopped being useful; the band of products it's the right answer for is just narrower than the default reach for it would suggest.

It remains a strong choice for marketing sites with ISR, content-heavy pages where SEO and first paint are the product, and teams that genuinely benefit from colocating server logic with React components. For fully static content sites that don't need to ship the full React runtime, Astro and similar frameworks tend to be a better fit, since islands of interactivity beat shipping the whole engine for pages that don't need it.

For an authenticated console with a real backend, none of those advantages apply. The cost of the framework's flexibility and its attack surface comes without the matching benefit.

## The Deeper Point

Most of the complexity in modern frontend stacks exists to solve problems that authenticated applications do not have. The default reaction to a new framework feature is "should this be adopted," when the better question is usually "does the problem this solves exist in this app."

For us the answer was no, repeatedly, until the only honest move was to stop paying for the answers. An enterprise console is browser-side rendering of data the backend already owns, served from a CDN, talking to an API. Once that's accepted, the simpler architecture is just what's left after the parts that weren't doing any real work get removed.
