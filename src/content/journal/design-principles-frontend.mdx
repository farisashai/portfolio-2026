Over the past year as the sole frontend engineer at Formal, I've had to design countless interfaces across the platform. The engineering side of building a good product isn't solved, but it is more straightforward: bug-free features, maintainable abstractions, performance you can reason about and test. As someone who never formally studied UI/UX/HCI, the visual look and feel was harder to quantify — beyond a rough sense that something is ugly and could be better.

As I've gotten faster at building the right shape for a feature, I've internalized design principles from customer feedback and from following designers I admire online. Design is a craft anyone can learn. Simple improvements compound into a product that feels professionally crafted — with care, personality, and delight.

## User Success

The primary metric of a good interface is **user success**: maximize delight, minimize frustration. The product should help users get as much out of it as possible with the features available. Surprise them with capabilities they didn't know they needed. Establish patterns that make navigation obvious. Be helpful in error cases. Minimize friction.

We achieve this through three design layers — **visual**, **interaction**, and **motion** — each with its own principles, all in service of the same goal.

## Visual Design

Visual design is what users see before they do anything. Layout, typography, color, spacing, hierarchy. It builds trust or creates doubt before a single click.

Know what you're optimizing for. Some surfaces exist to delight — onboarding, a launch moment, a first-run experience where you're telling a story and earning emotional buy-in. Others exist to sell powerful functionality: admin tools, security consoles, infrastructure dashboards. There the job is usability and trust first — signals that the system is stable, reliable, and secure before anyone cares about personality. Flair on a policy editor doesn't build confidence; consistent alignment and clear hierarchy do. Match the visual register to the job. Delight where the moment earns it. Default to restraint where the user is deciding whether to trust you with something important.

### Design Systems

Build consistent systems. An allowlisted spacing scale, a small set of display and label fonts, a fixed set of font sizes, and a component library that is auditable and tweakable without reproducing every edge case state.

The system should feel **considerate, differentiated yet restrained** — predictable consistency anywhere the user goes, that never slips. You don't need a monotone product or an infinite palette. You need a small set of values in an allowed set. Brand uniqueness lives in those constraints, not in one-off decoration.

### Alignment

Properly aligned interfaces don't feel special. They feel natural. Misaligned interfaces are a terrible distraction — they signal nobody cared, and they break trust even when the core feature is correct.

Alignment nudges the user's cognitive model of what belongs together. Proximity, shared borders, and consistent axes tell people which fields are one unit, which sections are separate, and what order to read — before they read a word of copy.

Alignment is broader than "things line up." It's where elements sit on the x and y axis inside a container — flex row, flex column, grid, or nested combinations. It's also **what gets grouped together**: related controls belong in an input group — shared border, internal dividers, one visual unit.

A payment form with `USD 49.00` in one amount field, expiry and CVV on separate rows, and city, state, and ZIP each full width forces users to parse unrelated strings. Group the currency prefix with the amount. Pair expiry and security code on one row. Put city, state, and ZIP in a shared input group.

- **Use input groups for related controls.** Currency + amount, expiry + CVV, city + state + ZIP.
- **Share axes inside a group.** Same height, internal dividers — not gaps between separate boxes.
- **Separate unrelated sections** — card details and billing address get their own labels and spacing.

### Visual Hierarchy

Not everything has to be a grid or a table. Not everything needs a single font size. Differentiation guides the eye from the most important starting point to the finer details — in a way where the feature explains itself.

If everything is bold, nothing is bold. Size, weight, color, and spacing should indicate importance:

- **Size**: larger text draws attention — use it for importance, not just headings
- **Weight**: bold for primary actions, regular for secondary, muted for tertiary
- **Color**: foreground for critical info, muted tones for supporting text, accent for CTAs
- **Spacing**: generous whitespace separates sections and reduces noise

Restrain decoration: varying text colors, fonts, or sizes for their own sake does not help. Reserve visual flair for special experiences — onboarding, one-time setup, or semantic concepts that genuinely need representation.

### Density and Whitespace

Density is a property of the UI, not a universal constant. A feature with a lot to configure needs room to breathe without becoming a wall of controls. A feature that is a single toggle to enable something new doesn't need a full-page layout with empty margins.

Whitespace is often wasted. Empty space is an opportunity for a more interesting layout — not an excuse to spread three fields across a viewport. Consider the maximum information you can realistically fit before needing to split it.

## The Interface Should Explain Itself

This sits between visual and interaction design, because it depends on both.

**Assume zero customer support.** An interface that doesn't explain itself is a failure. Apple's best design is discovered on its own — users explore, and the interface rewards that exploration. Eventually, if you label and document everything on the surface, you trade clarity for clutter.

Putting your UI on crutches is a weak solution. Tutorials, tooltips on every field, and a docs site that compensates for confusing flows mean the design failed. Documentation should extend capability, not patch over poor design. If a feature needs a walkthrough, redesign the feature.

The interface should speak before the user asks. Empty states explain what goes here. Placeholder text shows the expected format. Disabled buttons have tooltips explaining why they're disabled.

The test: watch someone use it for the first time with no guidance. Where do they hesitate? Where do they reach for help? Those moments are design failures.

## Interaction Design

Interaction design is what happens when users start doing things — clicking, typing, navigating, configuring. It's the logic of the experience: how information is revealed, how complexity is managed, how the interface responds to intent.

### Multimodal Inputs

Users don't interact one way. They drag, type, click buttons, scroll, write code, edit JSON config, use natural language, or speak. With AI in the mix, the surface area has exploded — the same action might be invokable through a form field, a chat prompt, or a programmatic API call.

The goal is to use the **fastest interface to project the user's intent into the system**. A power user might prefer raw config. A less technical user might prefer guided inputs. Support both where it matters, hide advanced modes behind progressive disclosure, and don't punish the majority with complexity they don't need.

Physical input modes still matter and have countless valid approaches:

- **Touch**: targets large enough to hit (44px minimum), no hover-dependent features, intentional gestures
- **Keyboard**: full tab navigation, logical focus order, discoverable shortcuts (`Cmd+K`, `/` for search), obvious focus states
- **Mouse**: hover feedback, click targets sized for precision, drag that doesn't lag

Can the feature be used with only a keyboard? Only touch? Any failure excludes users.

A filter interface might offer a visual query builder for beginners and a raw SQL input for power users. Let the user toggle between modes.

### Progressive Disclosure

Not everything needs to be shown immediately. If a surface is getting too dense, or a control isn't the first thing someone will reach for, or it carries built-in complexity that only matters in specific cases — defer it. Show what the user needs now. Make the rest discoverable when context demands it.

The failure modes sit at opposite ends. A 20-step wizard for something that could be three fields forces unnecessary pacing and hides the shape of the task. A single massive control panel dumps every option on screen at once — nothing breathes, nothing guides the eye, and the user has to parse a wall of controls before doing anything useful.

The target is somewhere in between: a panel that feels open and scannable, with related controls grouped, common paths obvious, and advanced or rare settings tucked behind a collapse, a tab, or a secondary screen. The user should see enough to understand what the feature does and complete the happy path without scrolling through configuration they'll never touch.

Configuration is stress. Every option creates cognitive load — understand it, evaluate it, choose a value, live with the consequence.

**The principle:** Default to sensible choices. Hide advanced configuration behind progressive disclosure. If a setting is needed by fewer than 20% of users, make it opt-in, not front-and-center.

**The nuance:** Power users will always want more control. That's fine. Support them through escape hatches, advanced modes, or raw config files. But don't punish the majority by forcing them to navigate complexity they don't need.

**When to disclose later:**
- Advanced tuning that most users never change
- Options that only apply after a prerequisite is enabled (e.g. webhook URL after webhooks are turned on)
- Destructive or irreversible actions that need context first
- Power-user modes — raw JSON, SQL, regex — behind a toggle from the guided UI

**When to keep visible:**
- Fields required for the primary action
- Choices that change what the rest of the form shows
- Status and feedback the user needs to trust the system is working

The test: can someone complete the most common task without expanding a single collapsed section? If not, you buried something essential. Can they see the full scope of the feature without feeling overwhelmed? If not, you showed too much too soon.

### Discovery and Hidden Power

Not every feature needs to be visible. Keyboard shortcuts, context menus, hover-revealed actions, and command palettes keep the default interface simple while rewarding exploration. When a power user discovers `Cmd+K` or that columns reorder on drag, that moment builds loyalty.

Delight is the reward for exploration. Consistency compounds trust — if swiping left deletes one list item, users expect it everywhere.

### Reduce Friction

User errors signal friction. Unclear error messages, irreversible actions without confirmation, hidden affordances, unnecessary steps, slow feedback loops — each one kills adoption. When a field only accepts values from a known set, don't make users guess — use a select, radio group, or segmented control instead of free text with buried helper copy.

Other friction patterns:
- **Irreversible actions** without confirmation or undo.
- **Unnecessary steps**: multi-step flows that could be a single screen.
- **Slow feedback loops**: waiting 5 seconds to see if a form submission succeeded.

Take feedback without ego. When a user struggles, that's a design problem regardless of how they articulate it. Read support tickets. Join sales calls. Watch real sessions. Fix friction ruthlessly.

## Motion Design

Motion is the layer that makes an interface feel alive. Without it, even a well-structured product feels static and mechanical. With too much of it, everything feels slow.

**Speed is a product feature.** Linear proved this — the architecture of features matters on the engineering side, but the felt speed of every interaction is a design decision too. The key tension is delight versus latency. Animation should make waiting feel shorter and transitions feel smoother, without ever making a user wait.

Everything is restraint.

### React Fast

High-frequency interactions must have zero perceived latency. Table row hovers, keystrokes in inputs, button press feedback — if the user moves faster than the UI responds, it feels broken.

- **Skeletons over blocking spinners** when content is loading. Show the shape of what's coming instead of a blank screen.
- **Shimmers in skeletons** signal that loading is active, not frozen.
- **Loading spinners** when an action is in flight and the user needs to know something is happening — save, delete, submit.
- **Optimistic updates** for validated actions: update the UI immediately, roll back gracefully on failure with a toast and coherent state.
- **Preload predictably** — prefetch on hover or viewport entry when the next navigation is likely. Don't speculatively load everything.

Use optimistic updates for form submissions where validation passed, CRUD on user-owned data, and actions unlikely to conflict. Wait for payment transactions, concurrent writes, and shared state mutations.

### Subtle Motion Tricks

Motion should communicate, not decorate.

- **Hover without transition on rapid movement** — instant feedback as the cursor crosses rows or list items, so detection of the current item never lags behind the pointer.
- **Motion blur scaled to distance** — larger transitions get slightly more easing; small state changes snap.
- **Staggered fade-in by importance** — primary content appears first, secondary details follow. Guides attention without blocking interaction.
- **Microanimations at 150–250ms** for low-frequency state changes — modal entry, success confirmation, panel expand. Fast enough to feel responsive, slow enough to read.

If a user isn't paying attention, they should be able to miss any single animation entirely. The collection of precise, restrained motion is what creates the feeling of craft.

Anti-pattern: animating every hover state. Reserve motion for moments where it clarifies a state change or guides attention. A 200ms fade on success is helpful. A 1s animation that blocks the next click is hostile.

## Designing with AI

AI changes the iteration loop for design, not just for code. You can ask an agent to generate three to five visual directions on the same feature and iterate on what you like or dislike in each one. Generation is cheap. Judgement is not.

### Two ways agents design — and what each misses

**By code.** Describe the feature, get back JSX and CSS. Fast, editable, shippable. But code-first generation optimizes for what compiles, not what reads. It'll produce a working form before it produces a form with the right density, hierarchy, or alignment. The tree is correct; the picture is wrong. You get locally reasonable decisions that don't compose into something that feels considered.

**By screenshots.** Paste a reference or ask for a mock from an image. Better for the static frame — spacing, color, layout at a glance. But a screenshot is one frame. It doesn't capture hover states, loading skeletons, optimistic rollback, staggered reveals, or what happens when the user tabs through with a keyboard. Motion and interaction patterns live in behavior over time; a PNG can't encode them. You'll still have to specify those separately, and that's where vague feedback fails.

Both paths struggle with **vision and judgement** — knowing what the whole should feel like, what to leave out, when something is close versus when it's off in a way that compounds. Expert designers are still better at this, not because agents can't render UI, but because they've built a vocabulary for discussing what's wrong and what to try next.

### Learn the language

The gap you can close is vocabulary. Agents respond to precision. "This feels uneven" gives them nothing actionable. "Left-align these labels in a two-column grid with `7rem` for the label column" gives them a constraint. "Make it pop" is useless. "Increase weight on the section title only — keep labels at regular weight and muted" is a direction.

The principles in this article are partly that language:

- Not "messy" — **labels and inputs aren't on a shared grid**
- Not "inconsistent" — **border radius drifts across controls that should share a token**
- Not "overwhelming" — **too many options visible before the user has context for them**
- Not "sluggish" — **row hover has a 150ms transition on a high-frequency interaction**
- Not "confusing error" — **every field returns the same generic message**

You don't need to become a designer. You need to name what you see in terms an engineer or an agent can execute: flex versus grid, which axis to align on, which elements share a type scale, what gets disclosed later, what must be instant versus what can ease in.

Critique is a skill that compounds. The more specifically you can say what's wrong, the faster iteration gets — with a teammate, with an agent, or with yourself six months from now revisiting the same screen. The agent proposes. You judge. Better judgement starts with better words.

## Before You Ship

- Does it feel fast? Profile it. Trim animations. Preload predictable paths.
- Is the hierarchy clear? Glance away, look back — does attention land in the right place?
- Does the interface explain itself? Would a first-time user need a tutorial?
- Is density right for this feature? Too sparse wastes space; too dense overwhelms.
- Are high-frequency interactions instant? Hover, click, type — no lag.
- Does rollback work? Optimistic updates that fail should leave the user somewhere coherent.
- Is motion restrained? Every animation should earn its milliseconds.
- What can go wrong, and how does the interface guide users back to safety?

---

Good product designers develop instincts for grouping, hierarchy, friction, and disclosure — not from credentials, but from building and critiquing real interfaces. Frontend engineers can learn the same lens. It matters more now: agents handle implementation; the bottleneck is judgment. Whether a layout groups related fields, whether friction lives in the error copy or the control choice, whether motion earns its milliseconds — that's what separates a shipped feature from a product that feels considered.

You don't need to become a designer. You need to name what you see in terms an engineer or an agent can execute. Ship, watch where users struggle, and iterate with better words.
