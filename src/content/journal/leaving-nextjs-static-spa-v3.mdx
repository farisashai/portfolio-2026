## Why I Switched

Three years ago, putting an authenticated React application on Next.js was the obvious choice. The deployment experience was excellent and server-rendered React still felt like the future. Earlier this year I migrated Formal's enterprise control plane onto a Vite-built static SPA, deleted the server runtime, and moved auth into the Go API.

The decision was deliberate. The goals driving it had been accumulating on the roadmap and eventually reached the point of needing to be addressed together: a security posture that didn't expose the next framework or platform CVE, expected performance gains from removing an intermediate runtime, and a self-hostable build artifact that customers could deploy in their own VPC. They pointed at the same architecture.

The principle the migration converged on, and the lens for the rest of the piece: every runtime actor in a system should have unavoidable responsibility. Next.js no longer had any in this codebase.

## What Next.js Was Running

By the time the migration started, the Next.js runtime in production was hosting:

- Auth server routes: login, callback, logout, all wired to the WorkOS server SDK.
- Middleware re-checking sessions before requests reached the API.
- JWT pre-verification.
- Feature-flag-driven layout rendering.
- Server-rendered page shells.
- Theme rendering.
- Image optimization.
- Request forwarding to the real backend.

That inventory is what the migration had to address.

## What an Authenticated Console Actually Needs

An enterprise console is a narrow shape of application. It fetches authenticated data, renders it with client-side interactivity, and mutates state through forms. The constraints that follow:

- Every byte is user-specific.
- Crawlers stop at the login page, so SEO doesn't apply.
- No anonymous traffic worth caching.
- The data is dynamic and per-tenant; ISR and edge caching have nothing to do.
- The critical path is authenticated data hydration. The user stares at a skeleton until the API returns regardless of how the HTML got delivered.

Orchestration across services lives in Formal's Go API where the data and auth already are. The frontend renders that data and submits changes back. It's a view onto a backend, not a participant in its architecture.

## The CVE Laundry List and Env Var Leaks

Application-code footguns are the day-to-day risk. The structural risk is what the framework and its host platform expose by existing at all. The cadence of Next.js vulnerabilities has become a significant overhead:

- **CVE-2025-66478**: A CVSS 10.0 RCE in React Server Components.
- **CVE-2026-23864 & CVE-2026-23869**: RSC denial-of-service advisories.
- **Directory Traversal & Cache Poisoning**: Repeated cycles of vulnerabilities in the framework's internal routing and caching layers.
- **Environment Variable Leakage**: The persistent risk where non-sensitive env vars are enumerated because the framework doesn't default to a private-first posture for its server-side configuration.

A static SPA removes the class entirely: no caching decision on the server, no `'use client'` boundary to police, no env var exposure surface, no per-route runtime to choose. Auth is a cookie.

## The Mirage of Dev Convenience

Next.js and Vercel sold a model of artificial simplicity by owning both the framework and the hosting. It felt like magic when a git push "just worked." But as projects scale, that convenience morphs into a scaling tax.

The modern reality is that DNS and cloud providers like Cloudflare, Netlify, Render, or AWS offer native monorepo support, branch previews, subdomains, and instant rollbacks at a fraction of the cost. They provide these as infrastructure primitives, not as a proprietary layer tied to a specific JavaScript runtime. Moving to a static SPA allowed us to use these native platform features without the arbitrary scaling markups or framework-locked hosting requirements.

## Network Topology: Ending the Double-Hop Disaster

Before I joined, the application suffered from an operational disaster where edge-routing and auth middleware doubled latency for months. Every request had to bounce through a Vercel Edge Function or a Node.js middleware layer just to verify a session before being forwarded to the actual Go backend.

This "double-hop" added 100-300ms of useless latency. By moving to a static SPA, we restored the optimal network topology: the browser talks directly to the Go API over a single multiplexed HTTP/2 connection. Across Formal's workloads, response times dropped 20 to 60 percent. The runtime was mostly adding topology, not capability.

## Client Experience and Hydration Hacks

Next.js introduces subtle UX frictions in professional tools. Prefetching can actually block page link clicks if the network is flaky or the server-side component is slow to respond. The infamous "hydration flash"—where the theme (light/dark mode) or auth state flickers—is a result of trying to synchronize server state with a client that already knows the truth.

In a static SPA, we handle theme state and auth checks natively on the client without hacks. Skeleton layouts are superior for perceived performance; users understand they are waiting for data from an API. Trying to cache those API requests at the server-framework layer (the Next.js fetch cache) is a complex, untested mess that often leads to stale data or, worse, cross-user data leaks.

## Compliance and Auditable Security

Keeping the frontend statically compiled means your security boundary is clean and auditable. Pushing JWT and auth verification strictly to the hardened Go gateway is clean and SOC-2 compliant. We no longer execute sensitive auth parsing logic in V8 or Node.js edge sandboxes where framework bugs could bypass our checks.

The frontend went from "auth participant" to "unauthenticated client that asks the API who the user is." Handing the repo to a stranger would only let them rebuild the product; they could not impersonate users or exfiltrate sessions.

## The Benefit of Static CDN Assets

A static frontend is just assets on a CDN. This means:
- **Zero compute scaling costs**: As our user base grows, our frontend serving cost remains effectively zero.
- **Infinite Scalability**: Uptime is tied to the CDN's global availability, not a Node.js process's health.
- **Legible Supply Chain**: Stripping Next.js and its server SDKs removed hundreds of transitive dependencies. What remains are primitives we trust: React, TanStack Router, and TanStack Query.

## Conclusion: Spending Your Innovation Tokens

Framework decisions are often made without a second thought, but they carry massive upstream architectural debt. I like to frame this through Dan McKinley’s "innovation tokens" philosophy. You only have a few tokens to spend on complex or risky technology.

Choosing a massive, opinionated meta-framework for a simple authenticated console is a waste of those tokens. Real simplicity is avoiding those headaches by choosing reliable, boring, and explicit architectures. Subtract the parts of the framework that weren't doing real work, and you're left with the speed, security, and sanity of a static SPA.
